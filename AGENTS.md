## Project preferences
- Use Hydra for runtime/training configuration; do not add a separate `config.py`.
- Prefer direct imports.
- Do not break tasks into subtasks unnecessarily.
- Unless explicitly requested, do not preserve backward compatibility; prefer the direct change over compatibility aliases or shims.

## Required implementation and review workflow

- Read the relevant `ROADMAP.md` ticket before changing code. When an implementation decision is ambiguous, use `PHILOSOPHY.md` as the decision policy instead of inventing a new project direction.
- Start and continuously update a draft pull request while implementing. The PR is the live handoff, not something written only after the work is finished. If PR creation is unavailable, prepare the complete PR body from `.github/pull_request_template.md` and state the blocker instead of inventing a PR URL.
- After implementation, run `/review` as an independent pass, separate from implementation, against `PHILOSOPHY.md`, the ticket acceptance criteria, and only the applicable sections of `CHECK.md`.
- If `/review` is unavailable or does not return a verdict, record the review as `blocked` and prepare the complete review handoff. Do not declare a passing verdict without a completed review.
- The post-implementation `CHECK.md` review is mandatory. It checks the real ML system, performance, DGX behavior, research integrity, and unnecessary complexity. It does not by itself require adding unit tests or generic software tests. Tests explicitly required by the roadmap ticket or needed to prove a mathematical/data invariant still remain in scope.
- If the review returns `FAIL`, do not mark the ticket complete. Record what failed and the evidence, hand off the failed review with the relevant repository context, constraints, and a concrete repair request, then repeat repair and independent `/review` until the result is `PASS` or an explicitly justified `PASS WITH NOTE`.
- Every PR must preserve its implementation, review, repair, and re-review outcomes, including important findings and evidence. Do not erase failed cycles.
- Record observable engineering rationale, changes, failures, evidence, and handoff context. Do not claim access to hidden chain-of-thought.
- Human review and merge is the default. An agent may self-merge only when a human has explicitly authorized that named PR or a bounded ticket/goal series, the authorization is recorded in the PR, and every guarded merge gate in `PHILOSOPHY.md` and `docs/agent-model-workflow.md` passes for the exact head commit. Tool access, authorship, or a passing self-review is never authorization. The independent `/review` verdict must be `PASS` or justified `PASS WITH NOTE`; all actionable findings must be repaired and independently re-reviewed; no blocking review decision, `CHANGES_REQUESTED` review, newer human objection, or unresolved review thread may remain; and an agent may not dismiss a human review to clear a gate. Inventory branch-protection required contexts and applicable configured workflows/checks, then require every expected check to be present and successful for the exact head; an absent, pending, skipped, cancelled, or otherwise non-successful expected check blocks merge. A no-check state is acceptable only with evidence that no check is required, configured, or expected. The PR must be current, conflict-free, and mergeable. Never bypass protections or self-merge work involving secrets or security controls, publication of private data, new paid resources, destructive or unrecoverable actions, unresolved legal/licensing questions, releases, deployments, account/permission changes, or another externally consequential non-routine action outside ordinary repository collaboration. Record the merging agent's final audit in the PR without changing the reviewed head. Immediately before merge, re-fetch and compare authorization, head, base, review decisions and objections, threads, expected checks and exact-head statuses, and mergeability; record the observation without changing the head, and abort for update, revalidation, or independent re-review on any drift.
- The bootstrap PR that introduces the guarded self-merge policy cannot authorize or merge itself. The preceding human-only rule governs it until a human merges it.
